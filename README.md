# 企业级会议室预约系统

帮助企业员工在浏览器里预约会议室，管理员维护房间和账号。一期面向**单一办公地点**，预约校验通过即生效，无需审批。

---

## 1. 项目简介

本系统替代微信群、Excel、口头预约等方式，让员工先选会议时间，再看到该时段可用的会议室，填写主题后一键预约。管理员可维护会议室信息、开通账号，并在必要时修改或取消任意预约。

**当前状态：** 需求与技术方案已完成，代码按 `specs/tasks.md` 拆分开发中。详见 [PROJECT_STATUS.md](./PROJECT_STATUS.md)。

---

## 2. 适合谁使用

| 角色 | 能做什么 |
|------|----------|
| **普通员工** | 登录、预约会议室、查看和取消自己的预约、看站内通知 |
| **管理员** | 以上全部 + 管理用户、会议室、全公司预约 |
| **开发/测试人员** | 按本文档启动项目，按任务文档实现与验收功能 |

用户**不能自己注册**，需管理员先创建账号。

---

## 3. 项目主要功能

### 员工（P0）

- 登录系统
- 预约会议室：选日期与时段（15 分钟一格）→ 选空闲房间 → 填会议主题 → 确认
- 我的预约：查看列表，取消尚未开始的会议

### 管理员（P0）

- 用户管理：创建账号、设为员工或管理员、重置密码
- 会议室管理：新增/编辑/删除房间，设置「维护中」
- 预约管理：查看、修改时间或房间、取消任意预约

### 站内通知（P1）

- 预约成功、自行取消、被管理员修改或取消时，在系统内收到消息

### 一期不包含

多地点、手机 App、审批、重复预约、邮件/钉钉推送、日程看板等（见 `specs/spec.md`）。

**出错时先看哪里：**

| 现象 | 建议查看 |
|------|----------|
| 页面打不开 | 前端是否启动、浏览器地址是否正确 |
| 登录失败 | 账号是否由管理员创建、后端是否启动 |
| 预约提交失败 | 页面提示文案；后端日志；是否时段冲突或房间维护中 |
| 接口 401/403 | 是否已登录、是否用管理员功能但角色不对 |

## 5. 本地运行方式

> **说明：** 模块 1 脚手架已完成。日常开发见下方「方式 B」；Docker 全栈见「方式 A」。

### 方式 A：Docker Compose（需已安装 Docker）

```bash
# 在项目根目录
docker compose up -d --build
```

| 服务 | 地址 |
|------|------|
| 前端 | http://localhost:5173 |
| 后端 API | http://localhost:8080/api/v1 |
| MySQL | localhost:3306（root / 123） |

### 方式 B：分别启动（开发常用）

**1. 数据库**

- 安装 MySQL 8（本机默认 `localhost:3306`，账号 `root`，密码 `123`）
- 库名 `meeting_booking`；Flyway 会在后端首次启动时自动建库建表

**2. 后端**

```bash
cd backend/meeting-booking-api
# 配置 application.yml 中的数据库连接后
mvn spring-boot:run
```

预期：API 基址 `http://localhost:8080/api/v1`（端口待确认）

**3. 前端**

```bash
cd frontend/meeting-booking-web
npm install
npm run dev
```

预期：开发服务器 `http://localhost:5173`；访问 `/` 会跳转到 `/login` 占位页

### 默认测试账号（种子数据，plan §10）

| 账号 | 说明 |
|------|------|
| `admin` | 管理员（密码见 Flyway 种子或环境说明，待脚手架完成后补充） |

## 9. 开发流程说明

1. 阅读 `specs/spec.md` 了解业务，阅读 `specs/plan.md` 了解接口与表结构。
2. 打开 `specs/tasks.md`，按模块顺序执行任务（先脚手架，再登录，再预约等）。
3. 每完成一条任务，在 `tasks.md` 中更新状态为「已完成」。
4. 用该模块文末的「页面测试 / 接口测试 / 异常测试」表格自测。
5. 同步更新 `PROJECT_STATUS.md` 中的已完成/未完成列表。

**给 AI Agent：** 一次只执行 `tasks.md` 中带明确文件路径的单条任务，并检查前置依赖是否已完成。

## 10. 如何测试功能是否正常

### 10.1 项目尚未跑起来时

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 打开 `PROJECT_STATUS.md` | 能看到当前阶段与模块列表 |
| 2 | 打开 `specs/tasks.md` | 能找到要做的下一条「未开始」任务 |

### 10.2 脚手架完成后（无业务功能）

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 启动后端 | 进程无报错；若有健康检查接口则返回成功 |
| 2 | 启动前端 | 浏览器能打开页面 |
| 3 | 打开 `/login` | 能看到登录表单 |

### 10.3 核心业务流程（全部开发完成后）

**员工预约**

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 用员工账号登录 | 进入系统，非管理员无管理菜单 |
| 2 | 打开「预约」页 `/book` | 能选日期、开始/结束时间（15 分钟一格） |
| 3 | 选未来 30 天内某时段，下一步 | 列出该时段空闲会议室 |
| 4 | 选一间房，填主题，提交 | 成功提示；「我的预约」中出现记录 |
| 5 | 打开通知（若已实现） | 有预约成功类消息 |

**员工取消**

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 打开「我的预约」 | 看到未开始的预约 |
| 2 | 点击取消并确认 | 状态变为已取消，该时段可被他人预约 |

**管理员**

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 用 admin 登录 | 能看到用户/会议室/预约管理入口 |
| 2 | 新建会议室 | 员工预约时可选（非维护中） |
| 3 | 将会议室设为维护中 | 员工预约流程中不可选或提交失败并有提示 |
| 4 | 创建员工账号 | 该账号可登录并预约 |

**冲突与异常（抽查）**

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 两人预约同一房间同一重叠时段 | 仅一人成功，另一人提示已被占用 |
| 2 | 预约过去的时间 | 无法提交，有明确提示 |
| 3 | 取消已开始的预约 | 不允许，有明确提示 |

更细的测试步骤见 `specs/tasks.md` 各模块末尾测试表。
